多租户架构与LiteLLM

概述​

LiteLLM 提供了一个集中化的解决方案，可以在多个租户之间扩展，使组织能够：

  • 集中管理 多个租户（组织、团队、部门）的LLM访问
  • 隔离成本和使用量 在不同的业务单元中
  • 委派管理 同时不牺牲安全性
  • 细粒度跟踪成本 （从组织到团队再到用户）
  • 无缝扩展 随着新团队和用户的加入
开源 vs. 企业版
  • 团队 + 虚拟密钥: ✅ 在开源版本中可用
  • 组织 + 组织管理员: ✨ 企业功能（获取7天试用）

您可以在开源版本中单独使用团队来实现多租户，或者在企业版中添加组织以获得额外的层级结构。

多租户挑战​

具有多租户架构的组织在部署LLM解决方案时面临以下挑战：

  1. 集中式 vs 分散式: 需要单一统一网关同时保持租户隔离
  2. 成本归因: 跟踪不同业务单元、部门或客户的支出
  3. 访问控制: 不同团队需要不同的模型、预算和速率限制
  4. 委派: 团队负责人应该管理自己的团队而不影响平台级管理员权限
  5. 可扩展性: 解决方案必须从10个用户扩展到10,000多个用户而无需架构变更

LiteLLM 如何解决多租户问题​

LiteLLM 实现了一个分层的多租户架构，共有四个层级：

1. 组织（顶级租户） ✨ 企业功能​

组织代表最高级别的租户隔离——通常是不同的业务单元、部门或客户。

  • 每个组织都有自己的：
    • 预算限制
    • 允许的模型
    • 管理用户（组织管理员）
    • 团队
    • 支出跟踪

用例：

  • 企业部门: 分开工程、营销和销售部门为不同的组织
  • 多客户SaaS: 每个客户都是一个独立的组织，具有完全隔离性
  • 地理区域: EMEA、APAC 和美洲作为单独的组织

关键功能：

  • 组织之间不能互相查看数据
  • 每个组织可以有多个团队
  • 组织管理员仅在其组织内管理团队
  • 在组织级别跟踪支出和使用量

组织API参考

2. 团队（中间层级分组） ✅ 开源版​

团队可以独立工作或位于组织内，代表逻辑上一起工作的用户群组。

提示

团队在开源版本中可用，并且可以在不使用组织的情况下作为主要的多租户边界。组织为企业部署提供了额外的层次结构。

  • 每个团队都有：
    • 团队特定的预算和速率限制
    • 管理成员的团队管理员
    • 共享资源的服务账户密钥
    • 模型访问控制
    • 细粒度的团队成员权限

用例：

  • 项目团队: ML研究团队、产品团队、数据科学团队
  • 客户子组: 客户组织内的不同部门
  • 环境分离: 开发、测试和生产团队

关键功能：

  • 团队继承了组织的约束（不能超出组织预算/模型）
  • 团队管理员可以管理自己的团队而不影响他人
  • 服务账户密钥在成员变更后仍然有效
  • 按团队跟踪支出并计费

团队API参考

3. 用户（个体成员） ✅ 开源版​

用户是属于团队并创建或使用API密钥的个人。

  • 每个用户可以：
    • 属于多个团队
    • 自己设置预算限制
    • 创建个人API密钥
    • 跟踪个人支出

用户类型：

  • 内部用户: 员工、开发人员、数据科学家
  • 团队管理员: 领导自己的团队，管理成员
  • 组织管理员: 管理其所在组织内的多个团队
  • 代理管理员: 平台级管理员

关键功能：

  • 个人支出单独跟踪
  • 用户可以同时属于多个团队
  • 基于角色的权限控制用户能做什么
  • 当用户被删除时，用户密钥也被删除

用户API参考

4. 虚拟密钥（认证层） ✅ 开源版​

虚拟密钥是用于认证请求和跟踪支出的API密钥。

每个密钥可以有三种类型：

密钥类型配置使用场景支出跟踪生命周期
用户专用user_id 仅此开发人员个人密钥用户级别删除与用户一起
团队服务账户team_id 仅此生产应用、CI/CD团队级别成员变更后仍然有效
用户 + 团队同时包含 user_id 和 team_id用户在团队上下文中的场景用户和团队级别删除与用户一起

示例场景：

  • 使用 用户专用密钥 供开发人员本地测试
  • 使用 团队服务账户密钥 供生产应用使用，即使员工离职也不会中断
  • 使用 用户 + 团队密钥 当您希望在团队预算内实现个人问责制

密钥API参考

基于角色的访问控制 (RBAC)​

LiteLLM 提供了跨层级的细粒度 RBAC：

全局代理角色（平台级）​

角色范围权限
代理管理员整个平台创建组织、团队和用户。查看所有支出。完全控制。
代理管理员查看者整个平台仅查看访问所有数据。不能做更改。
内部用户自有资源创建/删除自己的密钥。查看自己的支出。

组织/团队角色（受限制的）​

角色范围权限
组织管理员 ✨特定组织创建团队，添加用户，在其所在组织内仅查看组织支出。
团队管理员 ✨特定团队在其所在团队内管理成员、预算和密钥。

✨ = 付费功能

团队成员权限​

团队管理员可以为普通团队成员配置细粒度的权限：

  • 只读（默认）:
["/key/info", "/key/health"]


- 允许创建密钥:
```json
["/key/info", "/key/health", "/key/generate", "/key/update"]
  • 完整密钥管理:
["/key/info", "/key/health", "/key/generate", "/key/update", "/key/delete", "/key/regenerate", "/key/block", "/key/unblock"]

支出跟踪与成本归因​

LiteLLM 提供了多级支出跟踪，通过层级流动：

分层支出流​

组织支出
    ├── 团队1支出
    │   ├── 用户A支出
    │   │   ├── 密钥1支出
    │   │   └── 密钥2支出
    │   └── 服务账户支出
    │       └── 密钥3支出
    └── 团队2支出
        └── 用户B支出
            └── 密钥4支出

预算执行​

预算可以在每一层级设置并继承：

  1. 组织预算: $10,000/月
    • 团队1: $6,000/月（在组织限制内）
      • 用户A: $3,000/月（在团队限制内）
      • 用户B: $3,000/月（在团队限制内）
    • 团队2: $4,000/月（在组织限制内）

执行规则：

  • 团队预算不能超过组织预算
  • 用户预算不能超过团队预算
  • 当任何层级超出预算时阻止请求
  • 实时跟踪防止超支

常见的多租户模式​

模式1：企业部门​

场景: 大型企业需要多个部门集中访问LLM

企业设置（使用组织）:

平台 (LiteLLM 实例)
├── 工程部门组织 ✨
│   ├── 后端团队
│   ├── 前端团队
│   └── ML团队
├── 营销部门组织 ✨
│   ├── 内容团队
│   └── 分析团队
└── 销售部门组织 ✨
    ├── 销售运营团队
    └── 客户成功团队

开源替代方案（仅使用团队）:

平台 (LiteLLM 实例)
├── 工程后端团队
├── 工程前端团队
├── 工程ML团队
├── 营销内容团队
├── 营销分析团队
├── 销售运营团队
└── 客户成功团队

优势：

  • 每个部门/团队管理自己的预算
  • 部门负责人（组织/团队管理员）控制他们的团队
  • 中心化计费和模型访问
  • 跨部门成本可见性供财务使用

模式2：多客户SaaS​

场景: SaaS提供商为多个客户提供LLM功能

企业设置（使用组织）:

平台 (LiteLLM 实例)
├── 客户A组织 ✨
│   ├── 生产团队（服务账户）
│   ├── 开发团队
│   └── QA团队
├── 客户B组织 ✨
│   ├── 生产团队（服务账户）
│   └── 开发团队
└── 客户C组织 ✨
    └── 生产团队（服务账户）

开源替代方案（仅使用团队）:

平台 (LiteLLM 实例)
├── 客户A生产团队（服务账户）
├── 客户A开发团队
├── 客户AQA团队
├── 客户B生产团队（服务账户）
├── 客户B开发团队
└── 客户C生产团队（服务账户）

优势：

  • 完全隔离客户/团队之间
  • 每个客户/团队的计费和使用量跟踪
  • 客户/团队管理员可以自助服务
  • 生产服务账户密钥在员工离职后仍然有效

模式3：环境分离​

场景: 单一组织具有多个环境

平台 (LiteLLM 实例)
└── 公司组织
    ├── 生产团队
    │   └── 严格速率限制的服务账户密钥
    ├── 预发团队
    │   └── 中等限制的服务账户密钥
    └── 开发团队
        └── 测试用户密钥（用于测试的慷慨限制）

优势：

  • 每个环境都有独立预算
  • 不同模型访问（生产 vs. 开发）
  • 防止开发使用影响生产预算
  • 通过环境轻松归因成本

委派与自助服务​

LiteLLM 的一个主要优点是委派管理：

在没有 LiteLLM 时​

每个团队 → 请求平台管理员 → 管理员进行更改

❌ 平台团队瓶颈
❌ 缓慢的入职过程
❌ 不利于扩展

使用 LiteLLM 时​

代理管理员 → 创建组织 + 组织管理员
组织管理员 → 创建团队 + 团队管理员  
团队管理员 → 独立管理自己的团队

✅ 分散化管理
✅ 快速入职
✅ 扩展到数千用户

自助服务能力​

团队管理员可以：

  • 添加/删除团队成员
  • 为团队成员创建API密钥
  • 更新团队预算（在组织限制内）
  • 配置团队成员权限
  • 查看团队使用和支出情况

组织管理员可以：

  • 在其所在组织内创建新团队
  • 分配团队管理员
  • 查看组织范围内的支出
  • 管理跨团队的用户

平台管理员可以：

  • 创建组织
  • 分配组织管理员
  • 设置组织级策略
  • 查看全平台分析数据

可扩展性​

LiteLLM 的架构从小型团队扩展到企业部署：

小型团队（10-100名用户）​

  • 单个组织
  • 几个团队（5-10个）
  • 代理管理员管理一切

中等规模（100-1,000名用户）​

  • 多个组织
  • 许多团队（50多个）
  • 组织管理员委派给团队管理员

企业级（1,000多名用户）​

  • 很多组织（部门/区域）
  • 数百个团队
  • 完全委派的管理结构
  • 中心化可观测性和计费

关键可扩展性功能：

  • 随着规模增长无需架构变更
  • 基于数据库（PostgreSQL）以确保可靠性
  • 水平扩展支持
  • 有效的支出跟踪和日志记录

安全与隔离​

租户隔离​

每个租户（组织）都是隔离的：

  • ✅ 不能查看其他组织的数据
  • ✅ 不能访问其他组织的密钥
  • ✅ 不能超出其预算限制
  • ✅ 不能访问不在其允许列表中的模型

认证安全​

  • 平台管理员的主密钥
  • 具有范围权限的虚拟密钥
  • SSO集成支持
  • JWT认证
  • IP白名单

审计与合规性​

  • 所有的API调用都记录了用户/团队/组织上下文
  • 支持费用分摊和归因
  • 管理员操作被审计
  • 与可观测性工具集成

快速入门​

开源 vs. 企业版设置

以下步骤展示了完整的企业层级结构，包括组织。

对于开源版本，跳过第1步和第2步，并直接从第3步（创建团队）开始。

第一步：设置组织 ✨ 企业级​

创建第一个组织：

curl --location 'http://0.0.0.0:4000/organization/new' \
    --header 'Authorization: Bearer sk-1234' \
    --header 'Content-Type: application/json' \
    --data '{
        "organization_alias": "engineering_department",
        "models": ["gpt-4", "gpt-4o", "claude-3-5-sonnet"],
        "max_budget": 10000
    }'

第二步：添加组织管理员 ✨ 企业级​

curl -X POST 'http://0.0.0.0:4000/organization/member_add' \
    -H 'Authorization: Bearer sk-1234' \
    -H 'Content-Type: application/json' \
    -d '{
        "organization_id": "org-123",
        "member": {
            "role": "org_admin",
            "user_id": "admin@company.com"
        }
    }'

第三步：创建团队 ✅ 开源版​

企业级: 组织管理员在其组织内创建团队
开源版本: 代理管理员直接创建团队（不需要organization_id）

# 企业级: 组织管理员在他们的组织中创建团队
curl --location 'http://0.0.0.0:4000/team/new' \
    --header 'Authorization: Bearer sk-org-admin-key' \
    --header 'Content-Type: application/json' \
    --data '{
        "team_alias": "ml_team",
        "organization_id": "org-123",
        "max_budget": 5000
    }'

# 开源版本: 代理管理员直接创建团队
curl --location 'http://0.0.0.0:4000/team/new' \
    --header 'Authorization: Bearer sk-1234' \
    --header 'Content-Type: application/json' \
    --data '{
        "team_alias": "ml_team",
        "max_budget": 5000
    }'

第四步：添加团队管理员​

curl -X POST 'http://0.0.0.0:4000/team/member_add' \
    -H 'Authorization: Bearer sk-org-admin-key' \
    -H 'Content-Type: application/json' \
    -d '{
        "team_id": "team-456",
        "member": {
            "role": "admin",
            "user_id": "team-lead@company.com"
        }
    }'

第五步：团队管理员管理其团队​

# 团队管理员添加成员
curl -X POST 'http://0.0.0.0:4000/team/member_add' \
    -H 'Authorization: Bearer sk-team-admin-key' \
    -H 'Content-Type: application/json' \
    -d '{
        "team_id": "team-456",
        "member": {
            "role": "user",
            "user_id": "developer@company.com"
        }
    }'

# 团队管理员为成员创建密钥
curl --location 'http://0.0.0.0:4000/key/generate' \
    --header 'Authorization: Bearer sk-team-admin-key' \
    --header 'Content-Type: application/json' \
    --data '{
        "user_id": "developer@company.com",
        "team_id": "team-456"
    }'

用例示例​

示例1：费用分摊模型​

目标: 每个业务单元支付其自身的LLM使用费

设置:

  1. 根据分配的预算为每个业务单元创建组织
  2. 设置基于分配预算的预算
  3. 按组织跟踪支出
  4. 生成每月财务报告

结果: 财务部门可以准确地将费用分摊给各自的部门。

示例2：面向客户的AI产品​

目标: 向客户提供LLM功能，具有隔离和成本跟踪

设置:

  1. 按客户创建组织
  2. 使用服务账户密钥为生产工作负载提供支持
  3. 按客户组织跟踪支出
  4. 根据客户等级设定速率限制

结果: 准确计费给客户，防止“噪音邻居”，保持隔离。

示例3：开发 vs 生产环境​

目标: 通过不同的策略分离开发和生产环境

设置:

  1. 创建"开发"和"生产"团队
  2. 开发: 宽松预算、所有模型、用户密钥
  3. 生产: 独立预算、仅批准的模型，服务账户密钥
  4. 每个环境不同的速率限制

结果: 开发人员可以自由实验而不影响生产预算或可靠性。

最佳实践​

1. 组织设计​

  • ✅ 将组织映射到成本中心或客户
  • ✅ 设置现实的预算并留有增长缓冲
  • ✅ 每个组织分配1-2名组织管理员
  • ❌ 不要创建太多组织（增加管理开销）

2. 团队结构​

  • ✅ 保持团队与实际工作小组一致
  • ✅ 使用服务账户密钥进行生产
  • ✅ 给予团队管理员足够的权限以实现自助服务
  • ❌ 不要创建单用户团队（使用用户专用密钥代替）

3. 密钥管理​

  • ✅ 使用描述性密钥名称
  • ✅ 定期轮换密钥
  • ✅ 删除未使用的密钥
  • ✅ 根据用例选择合适的密钥类型
  • ❌ 不要在用户之间或团队之间共享密钥

4. 预算管理​

  • ✅ 在多个层级设置预算（组织 → 团队 → 用户）
  • ✅ 定期监控支出
  • ✅ 在预算耗尽前发出警报
  • ❌ 不要将预算设得太紧（可能会阻止合法使用）

5. 委派​

  • ✅ 对大型组织分配组织管理员
  • ✅ 对活跃团队分配团队管理员
  • ✅ 配置团队成员权限适当
  • ❌ 不要让每个人都成为代理管理员

常见问题​

Q: 用户可以属于多个团队吗? A: 是的，用户可以是多个团队的成员，并为每个团队拥有不同的密钥。

Q: 当用户离职时会发生什么? A: 仅针对用户的密钥被删除，但团队服务账户密钥仍然有效。

Q: 团队预算是否可以超过组织预算? A: 不可以，系统确保团队预算不能超出其组织的预算。

Q: 成本跟踪有多细粒度? A: 每个API调用都带有组织、团队、用户和密钥上下文进行跟踪。

Q: 我可以在没有组织的情况下创建团队吗? A: 可以！在开源版本中，团队可以独立工作而不需要使用组织。组织是企业功能，它在团队之上添加了额外的层级结构。

Q: 层级深度是否有上限? A: 层次结构为：组织 → 团队 → 用户 → 密钥（4层）。这涵盖了大多数用例。

Q: 如何从扁平结构迁移到分层结构? A: 你可以逐步创建组织和团队，然后将现有用户/密钥移入它们中。

总结​

LiteLLM 通过以下方式解决了多租户架构挑战：

  1. 分层结构: 组织 → 团队 → 用户 → 密钥
  2. 细粒度RBAC: 平台级和租户级别的角色
  3. 成本归因: 每个层级的支出跟踪
  4. 委派: 组织管理员和团队管理员自我管理
  5. 隔离: 强大的租户边界
  6. 可扩展性: 从10到10,000多个用户保持相同架构

开源 vs. 企业版​

开源版本 (团队 + 用户 + 密钥):

  • ✅ 团队作为主要的多租户边界
  • ✅ 团队管理员管理自己的团队
  • ✅ 带有团队/用户跟踪的服务账户密钥
  • ✅ 每个团队的预算和速率限制
  • ✅ 跟踪支出和日志记录

企业版 (增加了组织层):

  • ✨ 组织用于顶级租户隔离
  • ✨ 组织管理员管理多个团队
  • ✨ 组织级预算和模型访问
  • ✨ 层次化委派和报告

这使得LiteLLM适用于：

  • ✅ 多部门的企业
  • ✅ 为多个客户提供SaaS的提供商
  • ✅ 需要成本分摊/归因的组织
  • ✅ 需要自我服务LLM访问的团队
  • ✅ 任何多租户LLM部署
